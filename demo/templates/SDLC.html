Project Specification

Requirements Definition

The software engineering project is a web-based learning platform that allows users to create, manage, and take custom quizzes. This feature aims to enhance student engagement and facilitate self-directed learning by providing an interactive environment where quizzes can be easily created, stored, and accessed through a user-friendly interface.

The application enables users to log in, access flashcards, take multiple-choice quizzes on various subjects, and record their project progress, including the date, title, details of activities, challenges encountered, and solutions implemented. This task focuses on experience with Flask routing, SQLite database interactions, and basic web templating using Jinja, while also emphasising key security concepts, such as transitioning from storing plain-text passwords to implementing secure password hashing.

Functional Requirements

  • User Authentication: Implement signup/login/logout with an initial insecure approach (plain-text passwords in SQLite), then upgrade to secure (hashed passwords using Werkzeug). Use sessions to persist user state.
  • Database Setup: Create a SQLite database with tables for Users (id, username, password, email)
  • Progress Logs (id, user_id, date, title, details, image_path nullable)
  • Quiz (Preset quizzes for history and geography, alongside with a function to create new custom quizzes.)
  • Use SQL files for table creation.
  • Progress Logging of development process: A form to add logs (date, title, details) and a view to display user-specific logs, stored in the database.
  • Basic Quiz Interface: A simple frontend to display flashcards and multiple-choice quizzes for one subject.
  • Users can submit answers and see feedback

Non-functional requirements

  • The codebase should have a clear structure for learning purposes.
  • Be easy to run locally with minimal dependencies.
  • Lightweight, with no unnecessary packages.
  • Use Flask for backend routing and Jinja for templating (e.g., base.html for inheritance).
  • Demonstrate insecure vs. secure practices
  • Plain-text vs. hashed passwords; raw vs. parameterised queries in code and documentation.
  • Code must be version-controlled with Git (include repository link in submission).
  • App should handle basic errors (e.g., flash messages for invalid login)

Constraints

  • Use only Flask, sqlite3 (no SQLAlchemy), and built-in Python libraries (e.g., datetime).
  • Install Werkzeug for hashing.
  • Limit to one subject for quizzes in this prototype.
  • No PWA features (e.g., service workers) or voice/ML yet.

Acceptance Criteria

  • Users can register and login, authenticating into the app.
  • Users can successfully create a quiz, add multiple-choice questions and save them to the database.
  • Users can view a list of quizzes they have created, and quiz details display correctly.
  • Quiz-taking functionality works as expected, with correct answers recorded and feedback displayed immediately.
  • All pages render correctly on desktop and mobile devices, and navigation is consistent across the platform.
  • Security measures (password hashing, SQL parameterisation) are implemented and functional.
  • The application performs reliably under normal usage without errors or crashes.

Waterfall Model and Agile Method Uses

  • Waterfall Model: Used for initial planning, requirements gathering, and design phases to establish a clear project scope and structure.
  • Agile Methodology: Employed during development and testing phases, allowing for an iterative progress and regular feedback, with adaptability to changes and allowing for improvements based on testing outcomes with my PWA.
  • Waterfall provided a solid foundation at the start, while Agile enabled for flexibility and responsiveness during coding and refinement.

Determining Specifications

  • Simplicity: A minimal starter app focused on clarity for learning and extension.
  • Modularity: Separate templates and static files for easy modification.
  • Documentation: This SDLC doc makes the project purpose and design clear for contributors.

User Interface

  • Responsive design using HTML, CSS, and Jinja templating (e.g., base.html inheritance across all HTML).
  • Clear, readable fonts and color contrast for accessibility.
  • Forms for login, quiz creation, question input, and progress logging.
  • Interactive quiz interface showing questions, choices, and immediate feedback.
  • Navigation menus linking all major pages (home, quizzes, flashcards, progress logs).

Integration Specifications

  • Integration with SQLite for storing users, quizzes, and progress logs.
  • Ability to scale or extend database tables for multiple subjects or analytics.
  • Compatibility with Flask routing and session management for all interactive features.

Security—Werkzeug & Login Functionality to Prevent SQL Injection

  • User authentication is implemented using Flask and Werkzeug for secure storage of sensitive user data.
  • Passwords are hashed using generate_password_hash and verified with check_password_hash, preventing storage of plain-text passwords.
  • Parameterised SQL queries are used for all login and registration forms to prevent SQL injection attacks.
  • User input is never inserted into SQL statements as plain text; instead, placeholders (?) are used, and user-provided data is safely passed as query parameters.

Integrating & Testing Individual Components

Register Page

  • Registered a new user account with valid credentials.
  • Verified that the account details (username, email, etc.) were correctly stored in the database.
  • Checked that passwords are hashed using werkzeug security to prevent storing plaintext passwords.
  • Attempted to register with an existing username/email to ensure proper error handling.
  • Checked that invalid input (e.g., empty fields) prevented account creation.

Login Page

  • Attempted to login with correct username and password, testing for authentication.
  • Attempted to login with an incorrect password and verified that login is denied.
  • Verified that the hashed password in the database matches using check_password_hash.
  • Checked that a session is created upon successful login and destroyed on logout.
  • Ensured SQL injection attempts (e.g., '' OR 1=1 --') are prevented.

Posting Progress

  • Accessed the "view progress" page after logging in.
  • Verified that the page displays all progress entries associated with the logged-in user.
  • Checked that other users’ progress is not visible unless allowed.
  • Tested empty state (no progress posted yet) to ensure proper messaging.

View Progress

  • Loaded a quiz from a pre-made text file (history_quizzes.txt) to be displayed properly.
  • Verified that all questions and answers are read correctly.
  • Tested quiz completion to ensure the score is calculated correctly.
  • Checked for handling of malformed or missing quiz files.

  • Verified that all questions and answers are read correctly.

Results

System Testing & Validation

Testing For Security Risks — Bandit

Bandit is a security linter for Python code that identifies common security issues. It scans the codebase for potential vulnerabilities and provides recommendations for remediation. By running Bandit on my app.py, it allows for an automatic scan of the Flask project for common security vulnerabilities. Running Bandit allowed me to identify weak points in the codebase early and ensure safer handling of user input, sessions, and database interactions, such as identifying that the debug mode being set as true allows a potential injection point.

Evaluation

Overall, the project managed to successfully meet the initial specificiations and requirements outlined in the SDLC document. The core functionalities, including the user authentication system, quiz creation and management, flashcard functions and progress logging were implemented effectively. The next steps would be to further improve the user interface for better usuability, cleaning up the codebase for maintainability, and adding more features such as analytics and additional quiz subjects. Security measures were also successfully implemented, transitioning from insecure practices to more secure methods using password hashing and parameterised queries. Testing confirmed that the application performs reliably under normal usage without errors or crashes, fulfilling the acceptance criteria set.
